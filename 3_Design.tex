\section{\EdgeQuantum Design}
\label{sec:design}

In this section, we present the design of \EdgeQuantum, a tiered-memory quantum circuit simulator specifically optimized for resource-constrained edge devices. 
\EdgeQuantum overcomes the physical memory limitations of edge hardware not by simplifying the quantum circuit, but by implementing a hierarchical memory management system that treats NVMe storage as a direct extension of the GPU memory.
By utilizing Unified Virtual Memory (UVM) as a zero-copy staging buffer and implementing a 4-buffer asynchronous pipeline, \EdgeQuantum enables the simulation of state vectors up to 4\,TB (39 qubits) on devices with only 8\,GB of RAM.

\subsection{Overall Procedure}
\label{design_overall}

Figure~\ref{overall_architecture} shows the overall procedure of \EdgeQuantum.
\EdgeQuantum processes a quantum circuit simulation through dynamic resource profiling and a UVM-based storage pipeline.

\noindent\textbf{Resource Profiling and Mode Selection.}
When a quantum circuit simulation starts, \textit{Resource Profiler} calculates the required state vector size and compares it with the available memory capacity. Based on this comparison, \EdgeQuantum determines whether the state vector fits within the Unified Virtual Memory (UVM) space. If the state vector fits within the available memory, \EdgeQuantum operates in \textit{Native Mode}. In this mode, the \textit{GPU Compute Engine} allocates the entire state in memory and executes the simulation using \textit{cuStateVec}. If the state vector exceeds the memory capacity, \EdgeQuantum activates \textit{Tiered Mode}, which uses NVMe storage as an extension of the memory hierarchy. By selecting the execution path based on memory capacity, \EdgeQuantum uses available hardware resources and prevents out-of-memory (OOM) failures.

\noindent\textbf{Tiered Mode Execution.}
In Tiered Mode, \EdgeQuantum employs a 4-buffer pipeline within \textit{Unified Virtual Memory} to stream data between storage and the GPU. Unlike traditional out-of-core simulators that incur host-to-device transfer overheads, \EdgeQuantum establishes a zero-copy data path by leveraging the Unified Memory Architecture (UMA) of edge devices. The full state vector is partitioned into fixed-size chunks, and four UVM ring buffers (B0 to B3) serve as the staging area. During execution, the \textit{GPU Compute Engine} directly accesses the UVM buffers to retrieve the current state (\textit{Compute Data}), applies quantum gates, and writes back the result (\textit{Updated Data}). At the same time, the \textit{CPU Ping-Pong Compression Engine} handles the \textit{Offloading Data} between UVM and storage. It performs \textit{Async Read} operations to load upcoming chunks from the \textit{NVMe SSD} into UVM buffers and \textit{Async Write} operations to flush computed chunks back to storage. To address asymmetric read and write latency of consumer-grade TLC SSDs, the pipeline depth is set to four. This buffering allows the GPU to compute on one buffer while the CPU handles high-latency I/O for others, preventing pipeline stalls. The compression engine compresses chunks on the fly using a dual-file ping-pong layout, enabling \EdgeQuantum to simulate large quantum systems (e.g., 39 qubits) that exceed the physical storage capacity of the edge device.

\subsection{Tiered Memory Architecture with UVM}
\label{design_memory}



A fundamental challenge in edge computing is efficiently bridging the gap between high-speed GPU execution and low-speed storage.
On NVIDIA Jetson platforms, the CPU and GPU share a physical DRAM pool (Unified Memory Architecture), yet standard memory allocation APIs often fail to leverage this advantage for storage-backed applications.
Standard device memory (\texttt{cudaMalloc}) cannot be addressed by POSIX I/O system calls like \texttt{pread}, necessitating intermediate host buffers and explicit \texttt{cudaMemcpy}, which doubles memory usage and increases latency.
Conversely, Pinned Host memory (\texttt{cudaMallocHost}) supports direct I/O but suffers from poor cache performance on Tegra SoCs when accessed by the GPU.

\EdgeQuantum resolves this dilemma by exploiting Unified Virtual Memory (UVM).
We allocate pipeline buffers using \texttt{cudaMallocManaged}. Crucially, we utilize the \texttt{cudaMemAttachHost} flag during allocation.
This flag signals the CUDA driver that the memory pages should reside in the CPU's page tables by default.
This allows our custom I/O backend to perform \texttt{O\_DIRECT} file operations directly into the UVM buffer, bypassing the OS page cache.
When the GPU needs to process a chunk, we explicitly transition the buffer visibility using \texttt{cudaStreamAttachMemAsync(..., cudaMemAttachGlobal)}.
This operation updates the GPU's memory management unit (MMU) to map the physical pages already present in DRAM, enabling a true zero-copy data path where data read from the disk is immediately available for GPU computation.

\subsection{4-Buffer Asynchronous Pipeline}
\label{design_pipeline}

The most significant performance bottleneck in tiered-memory simulation is the high write latency of TLC NAND flash storage.
While read operations are relatively fast ($\approx$100\,ms per 256\,MB chunk), write operations can spike to 450\,ms once the SSD's SLC cache is exhausted.
A traditional double-buffering approach would stall the GPU, as the compute time ($\approx$220\,ms) is insufficient to hide this write latency.

To mitigate this, \EdgeQuantum implements a 4-buffer asynchronous pipeline.
The pipeline depth of 4 was mathematically determined to ensure that the write operation for a chunk $c$ has sufficient time to complete before its buffer is reused for chunk $c+4$.
The available time budget $T_{budget}$ in a 4-buffer system is defined as:
\begin{equation}
    T_{budget} = 3 \times (T_{read} + T_{compute})
\end{equation}
Given typical execution times on a Jetson Orin Nano ($T_{read} \approx 100$\,ms, $T_{compute} \approx 220$\,ms), the budget is approximately 960\,ms.
Since the worst-case write latency $T_{write}$ is around 450\,ms, the condition $T_{budget} > T_{write}$ is satisfied ($960 > 450$), preventing pipeline stalls.

The pipeline state machine processes chunk $k$ using buffer $B_{k \pmod 4}$ as follows:
\begin{enumerate}
    \item \textbf{Wait Write:} The main thread waits for the completion of any pending asynchronous write task on buffer $B_{k \pmod 4}$. Due to the 4-buffer design, this wait time is typically zero.
    \item \textbf{Serialized Read:} The system executes a synchronous \texttt{pread} to load chunk $k$. We ensure no other write task is active during this phase to avoid NVMe bandwidth contention.
    \item \textbf{Async Write Submission:} Immediately after reading, the system submits an asynchronous write task for the \textit{previous} chunk ($k-1$) residing in buffer $B_{(k-1) \pmod 4}$. This task runs on a background thread using \texttt{std::async}.
    \item \textbf{Compute:} The GPU kernel for chunk $k$ is launched. Since the GPU and the NVMe controller are independent hardware resources, this computation overlaps perfectly with the background write of chunk $k-1$.
\end{enumerate}

\subsection{LZ4 Ping-Pong Compression}
\label{design_compression}

\begin{algorithm}[t]
\caption{Dynamic Ping-Pong Compression Strategy}
\label{alg:pingpong}
\begin{algorithmic}[1]
\Require Gate Layer $L$, Total Chunks $M$
\Ensure Compressed state vector is updated for Layer $L+1$

\State $OffsetOut \gets 0$
\If{$L \bmod 2 == 0$}
    \State $FileIn \gets \mathrm{FileA}, \quad FileOut \gets \mathrm{FileB}$
    \State $IdxIn \gets \mathrm{IndexA}, \quad IdxOut \gets \mathrm{IndexB}$
\Else
    \State $FileIn \gets \mathrm{FileB}, \quad FileOut \gets \mathrm{FileA}$
    \State $IdxIn \gets \mathrm{IndexB}, \quad IdxOut \gets \mathrm{IndexA}$
\EndIf

\For{$c = 0$ \textbf{to} $M-1$}
    \State \textit{// 1. Variable-size Read \& Decompress}
    \State $OffsetIn \gets IdxIn[c]$
    \State $SizeIn \gets IdxIn[c+1] - OffsetIn$
    \State $CompBuf \gets \mathrm{AsyncRead}(FileIn, OffsetIn, SizeIn)$
    \State $UVMBuf \gets \mathrm{LZ4Decompress}(CompBuf)$
    \State \textit{// 2. Quantum Gate Simulation (Overlapped)}
    \State $\mathrm{GPUCompute}(UVMBuf)$
    \State \textit{// 3. Compress \& Append-only Write}
    \State $SizeOut \gets \mathrm{LZ4Compress}(UVMBuf, CompBuf)$
    \State $\mathrm{AsyncWrite}(FileOut, CompBuf, OffsetOut, SizeOut)$
    \State \textit{// 4. Update Index Table}
    \State $IdxOut[c] \gets OffsetOut$
    \State $OffsetOut \gets OffsetOut + SizeOut$
\EndFor
\State $IdxOut[M] \gets OffsetOut$ \Comment{Record total compressed size}

\end{algorithmic}
\end{algorithm}


For simulations where the state vector size exceeds the physical SSD capacity (e.g., $N \ge 35$ qubits requires $\ge 256$\,GB), raw storage becomes infeasible.
\EdgeQuantum addresses this by integrating a transparent compression layer using the LZ4 algorithm, chosen for its extremely high decompression throughput.
However, quantum gate operations pose a challenge for compression: modifying a state vector chunk changes its entropy and thus its compressed size, making in-place file updates impossible.

To solve this, we devised a "Ping-Pong" file strategy.
The system maintains two backing files, File A and File B.
During layer $i$, the system reads compressed chunks from File A, decompresses them into the UVM pipeline, applies the quantum gates, and then compresses the results to be appended to File B.
At the end of the layer, the roles of File A and File B are swapped.
To manage the variable-sized chunks, \EdgeQuantum maintains an in-memory \textit{Offset Table} that maps logical chunk indices to their physical byte offsets in the active compressed file.
This technique allows \EdgeQuantum to simulate a 39-qubit system (logical size 4\,TB) on a 256\,GB SSD by reducing the storage footprint to approximately 16\,GB for sparse state vectors.

\subsection{\EdgeQuantum Implementation}
\label{design_impl}

We implemented \EdgeQuantum as a standalone C++17 library with CUDA 11.4 support, extending the capabilities of the NVIDIA \textit{cuQuantum} SDK.
The implementation comprises approximately 1,200 lines of optimized code distributed across three core modules.

\noindent\textbf{1) Chunk Manager (\texttt{chunk\_manager.cpp}):}
This module abstracts the complexity of UVM allocation. 
It wraps \texttt{cudaMallocManaged} calls and exposes an API to toggle memory visibility.
Crucially, it manages the mapping between the logical global index of the state vector and the physical ring buffer index, ensuring data consistency across the pipeline.

\noindent\textbf{2) Pipeline Engine (\texttt{simulator.cpp}):}
This core module implements the 4-buffer state machine described in Section~\ref{design_pipeline}.
We utilize \texttt{std::future} and \texttt{std::async} to manage background I/O threads without the overhead of manual thread pool management.
The I/O backend leverages \texttt{O\_DIRECT} enabled \texttt{pread}/\texttt{pwrite} system calls to bypass the operating system's page cache. 
On Jetson's unified memory architecture, this ensures that data flows directly from the NVMe controller to the DRAM buffer used by the GPU, minimizing memory bandwidth consumption.

\noindent\textbf{3) Compression Layer (\texttt{compression.hpp}):}
This module implements the LZ4 ping-pong logic.
It encapsulates the complexity of variable-size storage, providing a fixed-size chunk interface to the upper layers of the simulator.
It handles the creation, swapping, and deletion of temporary backing files, ensuring that the storage footprint remains minimal throughout the simulation.
We open-source the code of \EdgeQuantum in the following link: \url{https://github.com/sunggonkim/IoTJ-EdgeQuantum}.
