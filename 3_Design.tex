\subsection{Overall Procedure}
\label{design_overall}

Figure~\ref{overall_architecture} shows the overall procedure of \EdgeQuantum.
\EdgeQuantum processes a quantum circuit simulation through dynamic resource profiling and a UVM-based storage pipeline.

\noindent\textbf{Resource Profiling and Mode Selection.}
When a quantum circuit simulation starts, \textit{Resource Profiler} calculates the required state vector size and compares it with the available memory capacity. To prevent mid-execution OOM crashes, this comparison uses 80\% of available GPU memory as a safety threshold, reserving headroom for kernel workspaces and runtime allocations. Based on this comparison, \EdgeQuantum determines whether the state vector fits within the Unified Virtual Memory (UVM) space. If the state vector fits within the available memory, \EdgeQuantum operates in \textit{Native Mode}. In this mode, \textit{GPU Compute Engine} allocates the entire state in memory and executes the simulation using \textit{cuStateVec}. If the state vector exceeds the memory capacity, \EdgeQuantum activates \textit{Tiered Mode}, which uses NVMe SSD as an extension of the memory hierarchy. By selecting the execution path based on memory capacity, \EdgeQuantum uses available hardware resources and prevents out-of-memory (OOM) failures.

\noindent\textbf{Tiered Mode Execution.}
In Tiered Mode, \EdgeQuantum employs a 4-buffer pipeline within \textit{Unified Virtual Memory} to stream data between storage and the GPU. Unlike traditional out-of-core simulators that incur host-to-device transfer overheads, \EdgeQuantum establishes a zero-copy data path by leveraging the Unified Memory Architecture (UMA) of edge devices. The full state vector is partitioned into fixed-size chunks, and four UVM ring buffers (B0 to B3) serve as the staging area. During execution, \textit{GPU Compute Engine} directly accesses the UVM buffers to retrieve the current state (\textit{Compute Data}), applies quantum gates, and writes back the result (\textit{Updated Data}). At the same time, \textit{CPU Ping-Pong Compression Engine} handles \textit{Offloading Data} between UVM and storage. It performs \textit{Async Read} operations to load upcoming chunks from \textit{NVMe SSD} into UVM buffers and \textit{Async Write} operations to flush computed chunks back to storage. To address the asymmetric read and write latency of storage, the pipeline depth is set to four. This buffering allows the GPU to compute on one buffer while the CPU handles high-latency I/O for others, preventing pipeline stalls. The compression engine compresses chunks on the fly using a dual-file ping-pong layout, enabling \EdgeQuantum to simulate large quantum systems (e.g., 39 qubits) that exceed the physical storage capacity of the edge device.


\subsection{Tiered Memory Architecture with UVM}
\label{design_memory}

A fundamental challenge in edge computing is bridging the gap between GPU execution and storage. On NVIDIA Jetson platforms, the CPU and GPU share a single DRAM pool due to the Unified Memory Architecture (UMA). Figure~\ref{uvm_diff} shows how \EdgeQuantum leverages this hardware characteristic to establish a more efficient data path compared to traditional approaches.

\noindent\textbf{Conventional Storage-backed Execution. }Despite the shared DRAM on UMA platforms like NVIDIA Jetson, conventional quantum simulation frameworks rely on standard device memory allocation models. As shown at the top of the figure, using standard device memory allocation (e.g., \texttt{cudaMalloc}) enforces a logical separation between host and device memory spaces. When retrieving state vector chunks from storage, POSIX I/O system calls (e.g., \texttt{pread}) cannot perform Direct Memory Access (DMA) into GPU VRAM. The system stages the data in an intermediate host-side buffer, followed by an explicit \texttt{cudaMemcpy} (H2D). This data path has two limitations for resource-constrained edge devices. First, it doubles the physical memory footprint (2x Memory constraint). Second, the additional memory copy operation saturates memory bandwidth and reduces simulation throughput.

\noindent\textbf{Zero-copy Execution for UMA. }To remove this bottleneck, \EdgeQuantum designs a zero-copy data path by controlling the visibility of Unified Virtual Memory (UVM) at the OS and driver level. As shown at the bottom of the figure, \EdgeQuantum allocates simulation buffers using \texttt{cudaMallocManaged} with the \texttt{cudaMemAttachHost} flag. This flag maps physical pages into the CPU page tables. With CPU-side visibility, \EdgeQuantum performs \texttt{O\_DIRECT} file operations, streaming state vector chunks directly into the UVM buffer while bypassing the Linux page cache and avoiding memory pollution. Before quantum gate execution, \EdgeQuantum issues a \texttt{cudaStreamAttachMemAsync} command with the \texttt{cudaMemAttachGlobal} flag. This asynchronous operation updates the GPU MMU page tables without data migration and transfers ownership of the physical DRAM pages to \textit{GPU Compute Engine}. By managing these UVM visibility transitions within a ring-buffer structure, \EdgeQuantum ensures that active data residency remains within physical DRAM limits, thereby preventing OS-level thrashing and removing explicit memory copies.
This approach uniquely exploits UVM attach-flag toggling to bypass the explicit memory copies required by conventional out-of-core simulators. \subsection{4-Buffer Asynchronous Pipeline}
\label{design_pipeline}


\begin{figure}[t]
    \centering
    % Ensure you have this file or use a placeholder
    \includegraphics[width=0.95\columnwidth]{Figures/edge_design3.3.png}
    \caption{Execution timeline of the asynchronous pipeline. The I/O serialization barrier (dotted line) strictly separates reads and writes to prevent NVMe bandwidth contention, while GPU computation perfectly overlaps with background storage eviction.}
    \label{design_3.3}
\end{figure}


The primary performance bottleneck in out-of-core quantum simulation on edge devices is the high and asymmetric I/O latency of commodity NVMe SSDs.
While sequential read operations can be serviced at 1.38~GB/s, write operations exhibit a lower throughput of 1.0~GB/s in Section~\ref{back2}. Traditional out-of-core simulators naively issue concurrent asynchronous reads and writes. Attempting such bidirectional I/O on edge storage causes bandwidth contention within the NVMe controller. This severely degrades overall throughput.

To mitigate this hardware limitation, \EdgeQuantum implements an asynchronous pipeline governed by an \textit{I/O Serialization Policy}. Figure~\ref{design_3.3} shows the execution timeline of this architecture. The core design principle is to isolate I/O directions to preserve NVMe bandwidth, while overlapping them with independent GPU computation. This isolation is enforced by an I/O serialization barrier, ensuring that a new read request is blocked until the preceding write operation completes. This 4-buffer configuration is selected to match the latency ratio between NVMe write operations ($T_{write} \approx 450ms$) and the combined read-compute window ($T_{read} + T_{compute}$), ensuring that a free UVM buffer is always available for the next $T_{read}$ phase. Driven by this policy, \EdgeQuantum processes state vector chunk $k$ through the following sequence:

\begin{enumerate}
\item \textbf{I/O Serialization Barrier:} To avoid bidirectional I/O contention, the host CPU halts execution at a synchronization barrier and waits for the background \texttt{pwrite} task of an earlier chunk (e.g., chunk $k-2$) to complete. This stalling prevents the OS block layer from mixing read and write queues and ensures that background write operations do not trigger internal garbage collection that would bottleneck the controller and delay the processing of the subsequent read request.

\item \textbf{Synchronous Read:} After clearing the barrier, \EdgeQuantum issues a synchronous \texttt{pread} system call to load chunk $k$ directly into the UVM staging buffer at sequential read bandwidth.
\item \textbf{Asynchronous Write:} After the read phase, the framework dispatches a background thread to asynchronously write the previously computed chunk ($k-1$) back to the storage.
\item \textbf{GPU Compute Overlap:} Concurrently with the background write task, \textit{GPU Compute Engine} executes quantum gate kernels on chunk $k$. Because the Jetson SoC has independent memory and NVMe controllers, this computation overlaps with the storage write. This hardware-level task separation ensures that CPU-intensive I/O management and GPU-intensive gate kernels operate without mutual interference.
\end{enumerate}

\noindent By serializing the I/O path and overlapping GPU execution, \EdgeQuantum hides write latency without causing bandwidth contention.



\subsection{LZ4 Ping-Pong Compression}
\label{design_compression}
\begin{algorithm}[t]
\caption{Dynamic Ping-Pong Compression Strategy}
\scriptsize
\label{alg:pingpong}
\begin{algorithmic}[1]
\Require Gate Layer $L$, Total Chunks $M$
\Ensure Compressed state vector is updated for Layer $L+1$

\State $OffsetOut \gets 0$
\If{$L \bmod 2 = 0$} \Comment{Swap files each layer}
    \State $FileIn, FileOut \gets$ FileA, FileB
    \State $IdxIn, IdxOut \gets$ IndexA, IndexB
\Else
    \State $FileIn, FileOut \gets$ FileB, FileA
    \State $IdxIn, IdxOut \gets$ IndexB, IndexA
\EndIf

\For{$c = 0$ \textbf{to} $M-1$}
    \State $OffsetIn \gets IdxIn[c]$ \Comment{Variable-size read}
    \State $SizeIn \gets IdxIn[c+1] - OffsetIn$
    \State $CompBuf \gets$ \textsc{AsyncRead}$(FileIn, OffsetIn, SizeIn)$
    \State $UVMBuf \gets$ \textsc{LZ4Decompress}$(CompBuf)$ \Comment{Decompress to UVM}
    \State \textsc{GPUCompute}$(UVMBuf)$ \Comment{Apply gates (overlapped)}
    \State $SizeOut \gets$ \textsc{LZ4Compress}$(UVMBuf, CompBuf)$ \Comment{Recompress in place}
    \State \textsc{AsyncWrite}$(FileOut, CompBuf, OffsetOut, SizeOut)$ \Comment{Append-only write}
    \State $IdxOut[c] \gets OffsetOut$ \Comment{Update index}
    \State $OffsetOut \gets OffsetOut + SizeOut$
\EndFor
\State $IdxOut[M] \gets OffsetOut$ \Comment{Record total size}

\end{algorithmic}
\end{algorithm}
 
For quantum simulations where the logical state vector size exceeds the physical capacity of the NVMe storage tier (e.g., $N \ge 35$ qubits requires $\ge 256$~GB), raw storage becomes a bottleneck. To extend simulation beyond physical storage limits, \EdgeQuantum integrates a compression layer using the LZ4 algorithm for high decompression throughput and low CPU overhead. However, quantum gate operations modify the state vector, which poses a challenge for compression schemes. Modifying a state vector chunk alters its entropy and results in unpredictable compressed sizes. This makes in-place file updates impossible and leads to \textit{entropy-driven file fragmentation}.

To address this entropy-driven fragmentation, we design \textit{Dynamic Ping-Pong Compression Strategy}, as detailed in Algorithm~\ref{alg:pingpong}. The strategy swaps the roles of two backing files, \textit{File A} and \textit{File B}, at the start of each gate layer (Lines 2-9) to enforce per-layer atomic updates of the compressed state vector. During the simulation loop (Lines 10-21), the framework executes a four-stage compressed pipeline for each chunk $c$. First, \EdgeQuantum retrieves the variable-sized compressed chunk from the source file using \textit{Index Table} ($IdxIn$) and decompresses it into the UVM staging area (Lines 11-14). Next, \textit{GPU Compute Engine} applies quantum gates to the UVM buffer while overlapping computation with I/O of adjacent chunks (Line 15). Then, the framework re-compresses the updated buffer and appends the compressed stream to the sink file (Lines 16-17). This append-only strategy avoids file fragmentation and maintains sequential write performance. Finally, the byte offset of the written chunk is recorded in the destination index table ($IdxOut$), maintaining the mapping for the next gate layer (Lines 18-20). This dual-file swapping technique allows \EdgeQuantum to simulate a 39-qubit system with a 4~TB logical footprint on a 256~GB SSD by reducing the storage requirement to approximately 16.9~GB, achieving a 242.7$\times$ compression ratio for sparse state vectors. This per-layer swap policy aligns physical storage residency with the logical boundaries of quantum circuit execution. It ensures that the state vector remains consistent throughout variable-sized compressed updates by treating each gate layer as a single atomic synchronization unit.


\subsection{\EdgeQuantum Implementation}\label{design_impl}
We implement \EdgeQuantum in C++17 and CUDA 11.4. The  1,200-line codebase is built on three core mechanisms: \texttt{cudaMemAttachHost}-based UVM visibility control for zero-copy UMA handover, \texttt{O\_DIRECT} enabled \texttt{pread}/\texttt{pwrite} for kernel page cache bypass, and a multi-threaded asynchronous pipeline for overlapping serialized I/O with GPU compute.
While optimized for the NVIDIA Jetson Orin Nano, \EdgeQuantum adheres to standard POSIX and CUDA Driver APIs, ensuring portability to other Tegra-based SoC architectures.
\EdgeQuantum provides a C++ API for integration into higher-level quantum simulation frameworks.
For validation (Section~\ref{sec:eval}), \EdgeQuantum utilizes \textit{cuStateVec} backend for core quantum operations, isolating the performance gains of our tiered memory architecture from underlying compute-engine optimizations.
The source code is publicly available at \url{https://github.com/sunggonkim/IoTJ-EdgeQuantum}.
