\subsection{GPU-Accelerated Quantum Circuit Simulation}
Quantum circuit simulation involves trade-offs between scalability and computational complexity~\cite{qsim, svsim, suzuki2021qulacs, villalonga2019flexible}. Tensor network contraction reduces memory usage, but its computational cost grows exponentially for highly entangled circuits~\cite{gray2018quimb, lykov2022tensor, pan2022simulation}. In contrast, full state vector simulation represents the quantum state as a complex vector of size $2^n$. It provides exact amplitudes without approximation for fidelity verification~\cite{guerreschi2020intel, de2007massively, qsim, suzuki2021qulacs}. To process these vectors, simulators like cuStateVec~\cite{bayraktar2023cuquantum} and Qiskit Aer~\cite{qiskit} rely on GPU acceleration. They allocate the state vector in VRAM and process the circuit layer by layer. CUDA kernels apply quantum gates in parallel. Because data resides on-device, updates avoid intermediate transfers.

However, GPU-accelerated simulation is limited by memory capacity. Memory footprint grows exponentially with the number of qubits at a complexity of $\mathcal{O}(2^n)$. Since each complex amplitude requires 8 bytes in single precision, a 30-qubit circuit demands 8GB. This fully exhausts the 8GB limit of edge devices (i.e., NVIDIA Jetson Orin Nano). Scaling further, a 34-qubit circuit requires 128GB and a 39-qubit circuit demands 4TB, exceeding memory capacities. As shown in Figure~\ref{Memory_wall}, this memory wall defines the capability of simulation frameworks in edge devices. For smaller qubit counts such as 27 qubits, GPU-native simulators like cuQuantum (Native) deliver extreme speedups of 0.53ms under QV circuits. Using unified virtual memory in cuQuantum (UVM) introduces overheads of 2.21ms. The workload still fits in memory. When scaling to 29 qubits with a 4GB footprint under QV circuits, both cuQuantum (Native) and cuQuantum (UVM) encounter out-of-memory (OOM) failures. This occurs because edge devices use a unified memory architecture. The 8GB physical memory is shared by the OS, CPU, and GPU. Allocating a 4GB state vector consumes most of the available GPU partition, leaving insufficient headroom for kernel workspaces and runtime allocations. To avoid mid-execution OOM crashes, simulators refuse execution when the state vector exceeds 80\% of available GPU memory, a safety threshold used for GPU workloads. 

To address this, out-of-core memory management is required. Frameworks such as BMQSim~\cite{zhang2025bmqsim} employ data compression and tiered memory pipelines to mitigate memory limits. However, these approaches are optimized for server-grade GPUs with discrete PCIe topologies and abundant host resources. Adapting them to edge environments requires falling back to blocking I/O on external NVMe storage. Their pipelines assume host-driven DMA paths that are absent on UMA SoCs. Such naive offloading stalls GPU computation and incurs severe I/O latency. We quantify this effect using \textit{BMQSim-like} baseline in Section~\ref{setup}.

\subsection{Memory Architecture in Edge Computing}\label{back2}

To handle data-intensive applications such as quantum simulation on edge devices, it is necessary to understand the underlying memory architecture. Unlike HPC clusters that use discrete CPUs and GPUs connected via PCIe, edge devices such as the NVIDIA Jetson Orin Nano use a System-on-Chip (SoC) design with a Unified Memory Architecture (UMA)~\cite{nvidia2022jetson}. In this architecture, the CPU and GPU share a single physical DRAM pool, eliminating PCIe data transfers. However, this integrated design introduces severe bandwidth asymmetry depending on the access path and storage tier.

\noindent\textbf{Unified Memory Architecture. }As shown in Figure~\ref{fig:bg_bw_dram}, within LPDDR5 DRAM, GPU-driven and UMA transfers (e.g., GPU D2D, UMA H2D/D2H) deliver about 25~GB/s, whereas host-driven CPU memory copies achieve only 6.0~GB/s. This is a 4$\times$ difference within the same physical memory. Recent studies in large-scale data processing propose out-of-core mechanisms~\cite{lin2022hm, hwang2020centaur} that use a user-space buffer manager to prefetch data and overlap I/O with computation. However, applying these techniques to quantum simulation on unified memory architectures has challenges. Existing frameworks assume separate memory spaces and rely on host-driven staging. This causes their data movement to be limited by the 6.0~GB/s CPU memcpy rate, failing to exploit the higher UMA bandwidth.

\noindent\textbf{External Storage Device. }As shown in Figure~\ref{fig:bg_bw_nvme}, when data moves to the external NVMe SSD, bandwidth decreases significantly. Sequential reads and writes provide about 1.38~GB/s and 1.0~GB/s. This is an 18$\times$ reduction compared to UMA transfers. Small-block random accesses reduce bandwidth further to 62~MB/s, which is a 400$\times$ reduction. Existing out-of-core approaches rely on operating system virtual memory management (OS-native swapping) to handle data larger than physical memory. This approach causes performance degradation due to the gap between the GPU driver and the OS file system. When a GPU kernel accesses a page that is not in memory, a page fault occurs and execution stops until the OS loads the data from disk. These page faults generate small-block storage accesses and force NVMe into the random read state (62~MB/s). The OS does not consider GPU access patterns. This leads to inefficient eviction decisions, I/O thrashing, and low GPU utilization. Beyond bandwidth, the physical storage capacity is limited to approximately 256~GB, which is insufficient to store state vectors with 35 qubits or more ($\geq 256$~GB).




Additionally, generic I/O libraries also do not consider quantum gate access patterns, where memory stride increases exponentially with the target qubit index. This results in inefficient I/O and poor cache locality. Optimizing quantum simulation on edge devices requires a specialized memory management strategy. The design must use zero-copy capabilities of UMA to sustain 25~GB/s bandwidth. It must enforce large sequential NVMe accesses (1.38~GB/s) to avoid the 62~MB/s random I/O penalty. \EdgeQuantum implements a UVM-aware asynchronous pipeline. It combines sequential I/O scheduling with zero-copy computation and maintains performance when the state vector exceeds physical memory capacity.
